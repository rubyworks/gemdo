What

POM stands for Project Object Model. POM provides a standardize methodology for storing information about a Ruby project. The system is easy to understand, easy to use, versitile and extensible.

The name, POM, was blantently co-opted from Maven. And though it serves much the same purpose for Ruby projects as Maven's POM does for Java projects, it is quite a bit eaiser to comprehend, doesn't use XML and certainly does not double as a build scripting language (a la Jelly). POM is simply the objectification of the common Ruby project, and no more.

Metadata

POM metadata is stored in a project subdirectory as a set of files. Each file stores a single piece of information. For example, in the above figures the meta/ directory listing of the POM project itself is given.

Each metadata file can be placed in a directory called meta/ or .meta/, depending on your preference and as a basis of separating distributable from non-distributable information.

Each file is read as a simple string, unless the file begins with "---", in which case it is parsed via YAML.

The use of such a file store might seem extravegant at first. After all, why not just use a single, easily editable YAML file? But the file store has distinct advantages that a single file simply cannot provide. Separate files can be easily manipulated and scripted, which is especailly useful for scaffold generators. They can also be lazy-loaded, improving performance when only select information in needed. And there's no need to be concerned of file layout --a YAML file, when rewritten by a script, can loose much of it's human readability. And, while multiple files may not be as easy to quickly edit in mass, consider that once a project has settled past the flurry of initial development, most metadata entries never change, so there's no need to continuely rewrite the file that contains it for the sake of other information.

Classes

The POM::Metadata class has the most common project attributes pre-defined, and has set typical defaults.

The POM::Project class encpsulates the Metadata class and a few other common project objects, such as the Manifest, as well as all the standard project locations as Pathname objects, like doc/ and lib/. This makes it very easy to work with a project as a Ruby object.

Future

POM is still evolving, but it has now been through enough prelimary usage to be a viable tool for the wider Ruby eco-system.
